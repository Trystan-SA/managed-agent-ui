# 10.0.0 — Connections (MCP Credentials & OAuth)

## 10.1.0 Overview

Connections let each user authorize Managed Agents to act on third-party MCP servers (Linear, Notion, GitHub, Atlassian, Sentry, Stripe, …) under the user's own identity. Credentials are stored in a **per-user Anthropic vault**, not in the local database — the app holds only a pointer to the vault and never sees the tokens after they're persisted.

Two auth methods are supported:

- **Static bearer** — the user pastes a personal API token. Preferred where possible because these tokens don't expire.
- **OAuth 2.0 (authorization code)** — admin registers OAuth client credentials once per service; users connect via redirect flow. Anthropic's vault handles refresh.

A curated service registry (`src/lib/mcp-registry.ts`) defines each integration's metadata, MCP server URL, and auth method. Custom (non-registry) MCP servers can also be connected by URL.

## 10.2.0 Data Ownership

| Data | Owner | Notes |
|------|-------|-------|
| MCP credentials (tokens, refresh blocks) | Anthropic (per-user vault) | App never persists the plaintext token — it POSTs once, then references by credential ID. |
| Vault pointer (`users.vault_id`) | Local DB | Lazily populated on first MCP action; recovered transparently if the vault was deleted or archived. |
| Service registry (MCP server URLs, auth type, instructions) | Code (`mcp-registry.ts`) | Static — adding a new service is a code change, not a DB change. |
| OAuth client credentials (client_id, client_secret, scopes override) | Local DB, AES-256-GCM encrypted | Admin-configured per service in Settings. |
| OAuth CSRF state | Local DB, short-lived | Deleted on consume; swept by expiry. |

## 10.3.0 Data Model

### 10.3.1 `users.vault_id`

Pointer to the user's Anthropic vault (`vlt_*`). Nullable — populated lazily by `getOrCreateUserVault()`. If the vault has been externally deleted or archived, the helper transparently mints a fresh one and updates the row. Callers can treat the returned ID as always-valid for the lifetime of the request.

The vault is created with `metadata.external_user_id = users.id` so an operator can trace a vault back to a local user in the Anthropic console even after a recreate.

### 10.3.2 `oauth_provider_configs`

One row per service that uses OAuth. Admin-only writes.

| Column | Type | Notes |
|--------|------|-------|
| service_id | text | PK. Matches a registry entry's `id`. |
| encrypted_client_id | bytea | AES-256-GCM |
| client_id_iv | bytea | |
| encrypted_client_secret | bytea | AES-256-GCM |
| client_secret_iv | bytea | |
| scopes | text | Optional override; NULL means use registry default. |
| updated_by | uuid | FK → users |
| created_at, updated_at | timestamp | |

### 10.3.3 `oauth_states`

Short-lived CSRF tokens for OAuth redirect flows. Created on `/api/connections/oauth/[serviceId]/start`, consumed on `/callback`. Indexed by `expires_at` for opportunistic cleanup.

| Column | Type | Notes |
|--------|------|-------|
| state | text | PK (random token echoed via the `state` query param) |
| user_id | uuid | FK → users (cascade) |
| service_id | text | |
| expires_at | timestamp | |
| created_at | timestamp | |

## 10.4.0 Service Registry

`src/lib/mcp-registry.ts` exports `MCP_SERVICES: McpServiceDefinition[]`. Each entry describes one integration:

| Field | Purpose |
|-------|---------|
| `id` | Stable identifier used in URLs and DB rows |
| `displayName`, `category` | UI labels |
| `mcpServerUrl` | The MCP endpoint the credential authenticates against |
| `authType` | `'static_bearer'` or `'oauth'` |
| `lifetimeNote` | User-facing note about token lifetime |
| `tokenUrl`, `instructions`, `tokenPlaceholder` | Static-bearer setup guidance (deep-link to provider's token page, step-by-step, placeholder) |
| `oauthConfig` | OAuth endpoints + default scopes + `tokenEndpointAuthType` (needed for refresh) |

Adding a new well-known service is a single registry entry + (for OAuth) one admin config row — no schema change.

## 10.5.0 API Routes

### 10.5.1 User-facing — Connections

| Method | Route | Description |
|--------|-------|-------------|
| GET | `/api/connections` | List registry services (each with `connected` status) plus any custom-URL credentials |
| POST | `/api/connections` | Connect a service. Body: `{ serviceId, token }` (registry) or `{ mcpServerUrl, displayName?, token }` (custom) |
| PATCH | `/api/connections` | Rotate an existing credential's token. Body: `{ credentialId, token }` |
| DELETE | `/api/connections?credentialId=…` | Remove a credential |

### 10.5.2 OAuth redirect flow

| Method | Route | Description |
|--------|-------|-------------|
| GET | `/api/connections/oauth/[serviceId]/start` | Mint CSRF state, redirect to provider's authorization URL |
| GET | `/api/connections/oauth/[serviceId]/callback` | Consume state, exchange code for tokens, store in vault, redirect to `/connections` |

### 10.5.3 Admin — OAuth provider configs

| Method | Route | Description |
|--------|-------|-------------|
| GET | `/api/admin/oauth-providers` | List configured providers (client IDs masked) |
| POST | `/api/admin/oauth-providers` | Upsert client_id + client_secret (+ optional scope override) for a service |
| DELETE | `/api/admin/oauth-providers?serviceId=…` | Remove a provider config |

### 10.5.4 Vaults (low-level)

Thin wrappers over the Anthropic vaults API for debugging / manual management. Not required by the normal `/connections` flow.

| Method | Route | Description |
|--------|-------|-------------|
| GET | `/api/vaults` | List the caller's vaults |
| POST | `/api/vaults` | Create a vault |
| GET | `/api/vaults/[id]` | Retrieve a vault |
| PATCH | `/api/vaults/[id]` | Update a vault |
| DELETE | `/api/vaults/[id]` | Archive a vault |
| GET | `/api/vaults/[id]/credentials` | List credentials in a vault |
| POST | `/api/vaults/[id]/credentials` | Add a credential |
| DELETE | `/api/vaults/[id]/credentials/[credentialId]` | Remove a credential |

## 10.8.0 Agent Integration

When an agent is configured with `mcp_servers`, the chat flow attaches the user's vault ID on session creation so Anthropic's runtime can resolve each server's credential. An MCP server on an agent is considered *usable* when the caller has a credential in their vault whose `mcp_server_url` matches the agent's server URL. `AgentMcpServers` surfaces this status on the agent inspect and dashboard chat forms; missing credentials link straight to `/connections`.

## 10.9.0 Design Decisions

1. **Credentials live in Anthropic's vault, not our DB.** We store a pointer and re-resolve on demand. This means a compromised local DB cannot leak third-party tokens, and we don't own refresh logic.
2. **Static bearer is preferred when available.** Personal API tokens don't expire, so there's no refresh surface at all — even for services that also offer OAuth (e.g., Linear appears twice in the registry: `linear` for PAT, `linear-oauth` for OAuth).
3. **OAuth client config is per-deployment, not per-user.** One admin-configured OAuth app serves every user of a given instance — matches how multi-tenant SaaS integrations are normally registered.
4. **Vault recovery is transparent.** If Anthropic archives a vault out from under us, the next request re-creates one — users don't see an error. The `external_user_id` metadata keeps the trail visible on Anthropic's side.
5. **Registry as code.** Service metadata (URLs, scopes, instructions) changes with deployments, not at runtime. Avoids a DB migration for every new integration.
