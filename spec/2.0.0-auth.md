## 2.1.0 User Roles

Two roles govern access:

| Role | Description |
|------|-------------|
| `admin` | First registered user. Can manage API keys, invite/remove users, configure SMTP. |
| `member` | Invited users. Can use agents, environments, sessions, and manage personal preferences. |

Role is stored in the `users.role` column (default: `member`).

## 2.2.0 Registration Flow (Admin Setup)

One-time setup â€” permanently disabled after the first user is created.

### 2.2.1 Steps

1. User navigates to `/register`
2. Server checks if any user exists â€” if yes, redirects to `/login`
3. User enters the setup password (from `SETUP_PASSWORD` env var)
4. **3-second artificial delay** after setup password submission to mitigate timing attacks
5. **Rate limit**: 5 requests per minute per IP to prevent brute-force attempts
6. On valid setup password, user proceeds to account creation (email, password)
7. Password hashed with bcrypt (12 salt rounds)
8. User created with `role: 'admin'`
9. Database-backed session created, cookie set, user redirected to `/dashboard`

### 2.2.2 Setup Password Gate

The registration page presents the setup password challenge **first**, before showing the account creation form. This two-step approach ensures:
- The setup password acts as an access gate to the registration form
- Brute-force protection via rate limiting (5 req/min) and artificial delay (3s)
- No account creation endpoint is exposed until the setup password is validated

## 2.3.0 Invite Flow (Member Onboarding)

Admins invite new users. Two invite modes:

### 2.3.1 Email Invite (SMTP configured)

1. Admin creates invite via `/api/admin/invites` with user's email
2. System generates a 48-hour invite token and sends email with invite URL
3. Invitee clicks link â†’ `/invite/[token]` â†’ sets their own password
4. Account created with `role: 'member'`, session started, redirected to `/dashboard`

### 2.3.2 Temporary Password Invite (no SMTP or manual)

1. Admin creates invite with a temporary password
2. Admin shares the invite URL manually (copied from UI)
3. Invitee visits `/invite/[token]` â†’ authenticated with temporary password
4. Account created with `role: 'member'` and `mustResetPassword: true`
5. User is immediately redirected to `/settings/reset-password` on every request until they set a new password

## 2.4.0 Login Flow

1. User submits email + password to `POST /api/auth/login`
2. Server verifies bcrypt hash
3. Database-backed session created, cookie set
4. Response includes `mustResetPassword` flag
5. Redirect to `/dashboard` (or `/settings/reset-password` if flag is set)

## 2.5.0 Password Reset Flow

For users flagged with `mustResetPassword` (e.g., after temporary password invite):

1. Middleware in `hooks.server.ts` intercepts all requests
2. If `mustResetPassword` is true, redirects to `/settings/reset-password` (except reset API, logout)
3. User submits new password (8+ characters) to `POST /api/auth/reset-password`
4. Password updated, `mustResetPassword` flag cleared
5. User can now access the full application

## 2.6.0 Session Management

- **Storage**: Database-backed (`auth_sessions` table) â€” survives server restarts
- **Token**: Random UUID per session
- **Cookie**: HTTP-only, Secure (in production), SameSite=Lax
- **Expiry**: 7-day max age, stored in `expires_at` column
- **Cleanup**: Expired sessions are piggyback-deleted during session lookups
- **Middleware**: `hooks.server.ts` checks cookie on every request, loads user ID, role, and `mustResetPassword` into `event.locals`

## 2.7.0 API Key Encryption

- **Algorithm**: AES-256-GCM
- **Key**: Derived from `ENCRYPTION_KEY` environment variable (64-char hex = 32 bytes)
- **Storage**: Encrypted key + IV stored in `api_keys` table
- **Access pattern**: Decrypted on-demand in `createAnthropicClient()` â€” never cached in plaintext
- **Scope**: Single global API key shared by all users, managed by admin

## 2.8.0 API Routes

| Method | Route | Auth | Description |
|--------|-------|------|-------------|
| POST | `/api/auth/verify-setup` | Public | Validate setup password (3s delay, 5 req/min rate limit) |

| POST | `/api/auth/login` | Public | Email/password login |

## 2.9.0 Route Protection

- **Public routes**: `/login`, `/register`, `/invite/[token]`, `/api/auth/*`
- **Auth-required routes**: All other routes require valid session cookie
- **Admin-only routes**: `/api/admin/*`, `/api/settings/api-key`, `/settings/admin`
- **Password reset intercept**: Users with `mustResetPassword` are confined to `/settings/reset-password`, the reset API, and logout
- `hooks.server.ts` middleware enforces all of the above

## 2.10.0 Security Decisions

| Decision | Rationale |
|----------|-----------|
| bcrypt over argon2 | Simpler dependency, sufficient for this scale |
| Database-backed sessions | Persist across server restarts; support multi-user without Redis |
| AES-256-GCM | Authenticated encryption â€” integrity + confidentiality for API keys and SMTP credentials |
| HTTP-only cookies | Prevents XSS from accessing session tokens |
| Server-side API proxy | API keys never reach the browser |
| Setup password gate with delay + rate limit | Prevents brute-force of one-time registration |
| Forced password reset | Temporary passwords from invites are never usable long-term |
| 48-hour invite expiry | Limits exposure window for invite tokens |
| Cascade deletes | Removing a user cleans up all their sessions, bookmarks, and preferences |
