# 2.0.0 — Authentication & Security

## 2.1.0 Registration Flow

One-time setup — permanently disabled after the first user is created.

### 2.1.1 Steps

1. User navigates to `/register`
2. Server checks if any user exists — if yes, redirects to `/login`
3. User enters the setup password (from `SETUP_PASSWORD` env var)
4. **3-second artificial delay** after setup password submission to mitigate timing attacks
5. **Rate limit**: 5 requests per minute per IP to prevent brute-force attempts
6. On valid setup password, user proceeds to account creation (email, password, Anthropic API key)
7. Password hashed with bcrypt (12 salt rounds)
8. API key encrypted with AES-256-GCM
9. Session cookie created, user redirected to `/dashboard`

### 2.1.2 Setup Password Gate

The registration page presents the setup password challenge **first**, before showing the account creation form. This two-step approach ensures:
- The setup password acts as an access gate to the registration form
- Brute-force protection via rate limiting (5 req/min) and artificial delay (3s)
- No account creation endpoint is exposed until the setup password is validated

## 2.2.0 Login Flow

1. User submits email + password to `POST /api/auth/login`
2. Server verifies bcrypt hash
3. Session created in memory, cookie set
4. Redirect to `/dashboard`

## 2.3.0 Session Management

- **Storage**: In-memory Map (acceptable for single-user deployment)
- **Cookie**: HTTP-only, Secure, SameSite=Lax
- **Expiry**: 7-day max age
- **Middleware**: `hooks.server.ts` checks cookie on every request, attaches user to `event.locals`

## 2.4.0 API Key Encryption

- **Algorithm**: AES-256-GCM
- **Key**: Derived from `ENCRYPTION_KEY` environment variable (64-char hex = 32 bytes)
- **Storage**: Encrypted key + IV + auth tag stored in `api_keys` table
- **Access pattern**: Decrypted on-demand in `createAnthropicClient(userId)` — never cached in plaintext

## 2.5.0 API Routes

| Method | Route | Description |
|--------|-------|-------------|
| POST | `/api/auth/verify-setup` | Validate setup password (3s delay, 5 req/min rate limit) |
| POST | `/api/auth/register` | Create admin account (rate limited, re-validates setup password) |
| POST | `/api/auth/login` | Email/password login |
| POST | `/api/auth/logout` | Destroy session |

## 2.6.0 Route Protection

- Public routes: `/login`, `/register`, `/api/auth/*`
- All other routes require valid session cookie
- `hooks.server.ts` middleware enforces this — redirects to `/login` on missing/invalid session

## 2.7.0 Security Decisions

| Decision | Rationale |
|----------|-----------|
| bcrypt over argon2 | Simpler dependency, sufficient for single-user |
| In-memory sessions | No Redis needed for single-user; sessions reset on restart |
| AES-256-GCM | Authenticated encryption — integrity + confidentiality for API keys |
| HTTP-only cookies | Prevents XSS from accessing session tokens |
| Server-side API proxy | API keys never reach the browser |
| Setup password gate with delay + rate limit | Prevents brute-force of one-time registration |
