# 11.0.0 — Scheduled Tasks (CRON)

## 11.1.0 Overview

Scheduled tasks run an agent on a cron-driven trigger, sending a rendered prompt template to a fresh or continuing session. The scheduler runs **in-process** in the SvelteKit server (`node-cron` + `cron-parser`) — there is no separate worker.

Each task belongs to a specific agent + environment pair, renders a prompt template, and records every execution (prompt sent, response, duration, error) for later inspection.

## 11.2.0 Data Model

### 11.2.1 `scheduled_tasks`

| Column | Type | Notes |
|--------|------|-------|
| id | uuid | PK |
| agent_id | text | Anthropic agent ID |
| environment_id | text | Anthropic environment ID — **required** at run time; selected at task creation so triggers don't inherit from the agent |
| prompt_template | text | Liquid-style template (see 11.4.0) |
| cron_expression | text | Standard 5-field cron |
| schedule_preset | text | Which preset the UI picked (`daily`, `weekly`, `every_1h`, …) — drives how the picker renders on re-open |
| timezone | text | IANA TZ; default `'UTC'` |
| session_mode | text | `'new_session'` (default) or `'continue_session'` |
| active_session_id | text | Populated when `session_mode = continue_session` — next run resumes this session unless it's terminated |
| enabled | boolean | Pause without deleting |
| locked_at | timestamp | Run lock — cleared on completion; locks older than 10 min treated as stale |
| next_run_at | timestamp | Authoritative next-run (computed from cron + tz) |
| last_run_at | timestamp | |
| run_count | integer | Monotonic |
| created_by, updated_by | uuid | FK → users |
| created_at, updated_at | timestamp | |

### 11.2.2 `task_executions`

One row per run. Inserted *before* validation so early failures (missing environment, etc.) still surface in the UI.

| Column | Type | Notes |
|--------|------|-------|
| id | uuid | PK |
| task_id | uuid | FK → scheduled_tasks (cascade) |
| session_id | text | Populated after session creation |
| status | text | `'running' | 'completed' | 'failed'` |
| prompt_sent | text | Rendered template (for audit) |
| response | text | Assistant's final message |
| error | text | When `status = failed` |
| started_at, completed_at | timestamp | |
| duration_ms | integer | |

### 11.2.3 `task_edit_history`

Append-only audit of task mutations. `changes` is a `jsonb` delta produced by the PUT handler.

| Column | Type | Notes |
|--------|------|-------|
| id | uuid | PK |
| task_id | uuid | FK → scheduled_tasks (cascade) |
| edited_by | uuid | FK → users (cascade) |
| changes | jsonb | `{ field: { from, to } }` shape |
| created_at | timestamp | |

## 11.4.0 Prompt Templates

Templates live in `prompt_template` and are rendered by `src/lib/server/template.ts` immediately before each run. The variable set is intentionally small:

| Variable | Example |
|----------|---------|
| `{{ timezone }}` | `Europe/Paris` |
| `{{ run_number }}` | `42` (1-indexed; equals `run_count + 1`) |
| Standard date helpers | e.g. `{{ now "YYYY-MM-DD" }}` — see `template.ts` for the full list |

The rendered prompt is persisted on the execution record (`prompt_sent`) so a run's inputs are reproducible.

## 11.5.0 Schedule Presets

`src/lib/schedule-presets.ts` defines the preset catalog the UI shows:

- `every_15m`, `every_30m`, `every_1h`, `every_2h`, `every_6h` — fixed cron, no parameters
- `daily` — user picks time (HH:MM)
- `weekdays` — user picks time; runs Mon–Fri
- `weekly` — user picks time + day of week
- `monthly` — user picks time + day of month

Custom cron isn't currently exposed in the UI — the presets cover the common cases and keep the editor approachable. `ScheduleCard` surfaces the picker + environment select + prompt editor on one row of the agent edit page.

## 11.6.0 Execution Pipeline

`src/lib/server/scheduler-executor.ts` (`executeTask(taskId)`):

1. **Lock** — atomic `UPDATE … SET locked_at = now() WHERE id = ? AND locked_at IS NULL`. If no row is updated, another worker has it; bail.
2. **Insert execution row up-front** — so validation failures (e.g. missing environment) still get logged.
3. **Validate** — reject if `environment_id` is null, with a human message telling the user to edit the schedule.
4. **Resolve session** — new session, or resume `active_session_id` if it's still live (otherwise fall back to new). Attaches the user's vault for MCP credentials.
5. **Send** — POST `user.message` with the rendered prompt.
6. **Collect** — stream the session's events; when `session.status_idle` fires, capture the assistant's final text.
7. **Finalize** — update the execution row (`completed` + `response` + `duration_ms`) or (`failed` + `error`); unlock the task; advance `next_run_at` via `computeNextRun`.

## 11.7.0 Scheduler Lifecycle

`src/lib/server/scheduler.ts` owns the in-process cron registry:

- `initScheduler()` — called once at server start; clears **stale** locks (older than 10 min) from crashed runs, then registers every enabled task's cron
- `addTask / updateTask / removeTask` — called by the CRUD handlers so edits take effect without a restart
- `triggerNow(taskId)` — used by `/run`
- `stopScheduler()` — graceful shutdown hook
- `computeNextRun(cronExpression, timezone)` — single source of truth for `next_run_at`; uses `cron-parser` with the task's IANA timezone

| Surface | Purpose |
|---------|---------|
| Agent edit page | `ScheduleCard` list — add, edit, remove schedules; each card has preset picker, time/day inputs, timezone, environment select, prompt template editor, session-mode radio |

## 11.9.0 Design Decisions

1. **In-process scheduler.** Matches the monolith architecture — no separate worker or queue service. Locks handle concurrency across replicas; stale-lock sweeping handles crashes.
2. **Environment pinned at task creation.** Earlier versions inherited the environment from the agent at run time, which meant changing an agent's default env silently changed every schedule. Binding `environment_id` on the task itself makes the schedule's target explicit and stable.
3. **Execution row inserted before validation.** Validation failures are the most common reason a user comes looking at execution history — the row has to exist.
4. **Preset-only in the UI.** Raw cron is powerful but a footgun; the presets cover daily/weekly/monthly/interval patterns which is almost everything users actually want.
5. **Edit history, not undo.** We record a diff per edit for audit rather than giving users a revert action — revert is a foot-gun against a concurrently-running task.
