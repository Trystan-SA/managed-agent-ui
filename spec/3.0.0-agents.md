# 3.0.0 — Agent Management

## 3.1.0 Overview

Agents are Claude model configurations stored in the Anthropic API. This domain provides CRUD operations, versioning, scheduled triggers, and local bookmarking for organization.

## 3.2.0 Data Model

### 3.2.1 Anthropic-Owned (Source of Truth)

Agents live in the Anthropic API with fields:

- `id`, `name`
- `model` — Claude model ID (or `{ id }` object for legacy entries)
- `system` — **authoritative** system prompt field (see 3.7.0)
- `description` — legacy mirror; new writes leave it null
- `tools` — tool configuration (see 3.8.0)
- `mcp_servers` — MCP server bindings (name + URL); see [10.0.0 Connections](./10.0.0-connections.md)
- `version` — monotonic; each edit creates a new version
- `archived_at` — soft-delete marker

### 3.2.2 Local Bookmarks

| Column | Type | Notes |
|--------|------|-------|
| id | uuid | PK |
| user_id | uuid | FK → users (cascade delete) |
| agent_id | text | Anthropic agent ID |
| nickname | text | Optional local name |
| pinned | boolean | Default false |
| created_at | timestamp | |

Bookmarks are **user-scoped** — each user has their own nicknames and pins. They add local metadata without duplicating Anthropic state. Deleting a user cascades to their bookmarks.

### 3.2.3 Scheduled Triggers

Agents can have any number of scheduled tasks attached — see [11.0.0 Scheduled Tasks](./11.0.0-scheduled-tasks.md). The agent inspect page shows a read-only summary; the edit page exposes full CRUD via `ScheduleCard`.

## 3.3.0 API Routes

| Method | Route | Description |
|--------|-------|-------------|
| GET | `/api/agents` | List all agents from Anthropic (global API key) |
| POST | `/api/agents` | Create new agent |
| GET | `/api/agents/[id]` | Retrieve single agent |
| PUT | `/api/agents/[id]` | Update agent (creates new version; nulls `description` to migrate legacy) |
| POST | `/api/agents/[id]/archive` | Archive agent |
| GET | `/api/agents/[id]/versions` | List version history |

Agent-related MCP status is served by `/api/connections` (see 10.5.1).

## 3.4.0 Pages

| Route | Purpose |
|-------|---------|
| `/agents` | List all agents with status + model + description (line-clamped), and MCP-status glance |
| `/agents/new` | Create agent form (name, model, system prompt, tools, MCP servers, schedules) |
| `/agents/[id]` | Inspect view: config summary, MCP status, read-only schedule list |
| `/agents/[id]/edit` | Edit form with sticky version-history panel (see 3.5.0) |

## 3.5.0 Versioning

Every update to an agent creates a new version in the Anthropic API. The version history endpoint returns all past versions.

### 3.5.1 Version History Panel

Version history lives on the **edit page**, not the inspect page, as a sticky right-hand panel. Clicking a version repopulates the form fields (name, model, system prompt, tools, mcp_servers) in place so a prior config can be inspected and resubmitted without leaving the page. Nothing is written until the user saves.

On save the edit page shows a success toast (see [9.0.0 § Toasts](./9.0.0-design-system.md)) and redirects to the inspect page.

## 3.6.0 Agent Configuration

- **Model**: Claude model ID (e.g. `claude-sonnet-4-6`, `claude-opus-4-6`, `claude-haiku-4-5`)
- **System prompt**: instructions, written to the `system` field
- **Tools**: built-in toolset + per-tool enable flags (see 3.8.0)
- **MCP servers**: name + URL bindings; an "enabled" server is one with a corresponding `mcp_toolset` entry in `tools`
- **Scheduled tasks**: see 11.0.0

## 3.7.0 `system` vs `description`

Early versions of this project wrote the system prompt into Anthropic's `description` field. That was a bug — `description` is meant for human-facing metadata, not the runtime system prompt. Current behavior:

- **Writes** (create + edit): always go to `system`. On edit, `description` is explicitly nulled to migrate legacy rows.
- **Reads**: prefer `system`; fall back to `description` when `system` is empty (covers agents created before the fix).

## 3.8.0 Built-in Toolset

The default toolset (`agent_toolset_20260401`) groups the core Anthropic tools; each can be individually enabled/disabled:

`bash`, `read`, `write`, `edit`, `glob`, `grep`, `web_fetch`, `web_search`

An absent config entry is treated as enabled. MCP tools are handled separately via `mcp_toolset` entries tied to `mcp_servers`.
