# 1.0.0 — Architecture & Overview

## 1.1.0 System Purpose

A self-hosted, multi-user dashboard for orchestrating Anthropic Managed Agents. Provides CRUD management for agents, environments, and sessions, plus a real-time chat interface with SSE streaming. The first registered account becomes admin; additional users are added via invite.

## 1.2.0 Architecture Pattern

**Monolithic SvelteKit application** — a single process serves both the SSR/CSR frontend and REST API routes. No microservices, no BFF pattern.

```
┌─────────────────────────────────────────────┐
│  Docker Compose                             │
│  ┌──────────────────────┐  ┌─────────────┐ │
│  │  SvelteKit App       │  │  Postgres   │ │
│  │  (Node adapter)      │──│  5432       │ │
│  │  Port 3000           │  └─────────────┘ │
│  │                      │                   │
│  │  Server routes ──────│──► Anthropic API  │
│  │  Pages/UI            │    (proxied)      │
│  │                      │──► SMTP (optional)│
│  └──────────────────────┘                   │
└─────────────────────────────────────────────┘
```

### 1.2.1 Rationale

- Multi-user with role-based access (admin/member) — single deployment serves a team.
- SvelteKit's server routes provide type-safe API endpoints colocated with their consumers.
- Docker Compose bundles both services for one-command deployment.
- Optional SMTP integration for email-based invites; falls back to manual invite URLs.

## 1.4.0 Data Ownership

**Anthropic API is the source of truth** for agents, environments, and sessions. The local PostgreSQL database stores:
- User authentication (email, hashed password, role)
- Database-backed auth sessions (token, expiry)
- Encrypted global API key (shared across all users, admin-managed)
- User invites (token, expiry, optional temporary password)
- Bookmarks (nicknames, pins) — local metadata over Anthropic resources, user-scoped
- User preferences (theme, defaults)
- SMTP settings (encrypted, for invite emails)

Local DB never caches or duplicates Anthropic API state.

## 1.5.0 Deployment

### 1.5.1 Production

- Multi-stage Docker build (builder → production) via `docker/Dockerfile`
- `docker/docker-entrypoint.sh` runs Drizzle migrations then starts the app
- Health checks for Postgres readiness before app starts
- Persistent `pgdata` volume for database storage
- `docker compose --env-file .env -f docker/docker-compose.yml up` — runs on port 3000

### 1.5.2 Development

- Single-stage `docker/Dockerfile.dev` — installs deps, runs Vite dev server
- `docker/docker-compose.dev.yml` — volume-mounts `src/`, `static/`, config files, and migrations
- Vite HMR with filesystem polling (1s interval) for Docker volume compatibility
- Postgres exposed on port 5432 for direct access
- `docker compose --env-file .env -f docker/docker-compose.dev.yml up --build` — runs on port 5173
- Separate `pgdata_dev` volume to isolate dev data from production

## 1.6.0 CI/CD

GitHub Actions pipeline (`.github/workflows/ci.yml`) runs on push and PR to `main`:

1. **Lint** — `npm run lint` (ESLint with TypeScript + Svelte plugins)
2. **Type check** — `npm run check` (svelte-check + TypeScript)
3. **Test** — `npm run test` (Vitest)

Run locally with `npm run ci`.
