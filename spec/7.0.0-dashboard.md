# 7.0.0 — Dashboard

## 7.1.0 Overview

The dashboard is the app's **primary chat surface**. It combines a session list sidebar with an embedded `ChatView` and a new-chat form so the user can pick up an existing session, start a new one, or review recent activity without leaving the page.

Since the Anthropic API key is global (shared by all users), every user sees the same resource catalog (agents, environments, sessions); only local preferences and bookmarks differ.

## 7.2.0 Page

| Route | Purpose |
|-------|---------|
| `/dashboard` | Primary chat surface + session list + new-chat form |

The standalone `/chat` listing has been folded into `/dashboard` — there is no separate index page.

## 7.3.0 Layout

```
┌───────────────┬──────────────────────────────────────────┐
│ Sessions      │  Header (selected session or new-chat)   │
│ ──────────    │ ──────────────────────────────────────── │
│ ● Active…     │                                          │
│ ● Active…     │  ChatView                                │
│ ────          │  (transcript + composer / working panel) │
│ ○ Past…       │                                          │
│ ○ Past…       │                                          │
│               │                                          │
│ [+ New chat]  │                                          │
└───────────────┴──────────────────────────────────────────┘
```

- **Sidebar** — two sections: active (status `running`) and past (everything else). Each row shows title, badge, and agent name. The new-chat button is pinned at the bottom.
- **Main pane** — either `ChatView` mounted against the selected session, or the new-chat form (agent + environment picker, with MCP status readout — see 7.5.0).
- Selecting a session swaps the main pane; creating a new session promotes the new-chat form into a live `ChatView` in place.

## 7.4.0 Data Source

All data is fetched from the Anthropic API at page load via server-side `+page.server.ts` using the global API key. No local caching — always reflects current state. If no API key is configured, the dashboard displays a gated state prompting an admin to add one in Settings.

### 7.4.1 Performance

The server load parallelizes the three Anthropic list calls with `Promise.all` and **does not auto-paginate** — a single page each:

- 50 sessions
- 100 agents
- 100 environments

Earlier versions walked every page serially, which dominated time-to-first-byte on workspaces with long history. The caps match the dashboard's glance-level use case; deeper drill-downs go through the domain-specific pages which paginate as needed.

## 7.5.0 MCP Status on the New-chat Form

When the user picks an agent, the new-chat form reads `/api/connections` and shows one row per MCP server the agent references: green "connected" or an "action needed" badge with a link to `/connections`. This catches missing credentials *before* the session is created rather than surfacing them mid-run. See [10.0.0](./10.0.0-connections.md).

## 7.6.0 Components

## 7.7.0 Design Decisions

1. **Chat *is* the dashboard.** Most sessions are opened shortly after creation; burying the composer one click deeper was friction without a benefit. Merging the two surfaces also lets the session list double as "recent activity."
2. **Single-page load, no pagination.** The dashboard is a glance view — 50/100 items is enough for orientation. Deeper views use `/sessions`, `/agents`, `/environments`.
3. **MCP status at the new-chat step.** A missing credential mid-run is a confusing failure; surfacing it on the picker keeps it in a context the user can fix immediately.
