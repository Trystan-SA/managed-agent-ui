# 9.0.0 — Design System

## 9.1.0 Overview

Custom SCSS design system built on CSS custom properties (design tokens). Supports dark and light themes with component-level style modules.

## 9.2.0 Architecture

```
src/styles/
├── _tokens.scss        # CSS custom properties
├── _mixins.scss        # Reusable SCSS mixins
├── _reset.scss         # Minimal CSS reset
├── _layout.scss        # Global layout (nav, page shell, grid)
├── global.scss         # Imports all partials
└── components/
    ├── _chat.scss      # Chat messages, composer, working panel, auto-scroll
    ├── _event-card.scss # Tool/thinking cards
    ├── _form.scss      # Inputs, labels, buttons
    ├── _table.scss     # Data tables
    ├── _badge.scss     # Status indicators
    ├── _sidebar.scss   # Session list sidebar
    └── _code-block.scss # Code display with copy
```

All component partials are imported from `global.scss`. An earlier revision omitted these imports — the component-level CSS existed as files but never shipped. The fix is load-bearing: compiled CSS jumped from ~200 to ~1400 lines once wired up. Any new component partial must be added to `global.scss` or it won't be included.

## 9.3.0 Design Tokens

### 9.3.1 Colors

Base palette (light theme — the default):

| Token | Value | Purpose |
|-------|-------|---------|
| `--color-bg` | `#f8f9fb` | Page background |
| `--color-surface` | `#ffffff` | Card/panel surfaces |
| `--color-surface-2` | `#f1f3f5` | Subtle surface variant |
| `--color-border` | `#e2e4e9` | Borders and dividers |
| `--color-text` | `#1a1a2e` | Primary text |
| `--color-text-dim` | `#5a5a74` | Secondary text |
| `--color-text-muted` | `#8888a4` | Tertiary/placeholder text |
| `--color-primary` | `#6366f1` | Primary actions (indigo) |
| `--color-success` | `#22c55e` | Success states (green) |
| `--color-warning` | `#f59e0b` | Warning states (amber) |
| `--color-danger` | `#ef4444` | Destructive actions (red) |
| `--color-info` | `#14b8a6` | Informational (teal) |

Dark theme overrides invert surfaces/text via `[data-theme="dark"]`.

### 9.3.2 Spacing Scale

`--space-1` (2px) through `--space-12` (64px) — consistent spacing scale.

### 9.3.3 Typography

- **Sans**: Inter
- **Mono**: JetBrains Mono
- Type scale via `--text-xs` through `--text-2xl`

### 9.3.4 Other Tokens

- Border radii: `--radius-sm` (4px) to `--radius-full` (9999px)
- Shadows: `--shadow-sm`, `--shadow-md`, `--shadow-lg`
- Surface colors, border colors, text colors — all tokenized

## 9.4.0 Theming

- **Light theme is the default** (base tokens define the light palette)
- Dark theme via `[data-theme="dark"]` selector overrides
- All components use tokens — no hardcoded colors
- Theme toggle persisted in `user_preferences` table
- New users get light theme on first login

## 9.5.0 Mixins

Reusable patterns in `_mixins.scss`:
- **Card**: Consistent card surfaces with border + shadow
- **Input**: Form input styling
- **Button**: Button variants (primary, secondary, danger)
- **Truncate**: Text overflow ellipsis
- **Responsive**: Breakpoint media queries

## 9.6.0 Component Styles

One SCSS partial per component type. Components `@use` only the tokens and mixins they need — no global style leakage.

## 9.8.0 Design Principles

- **DRY**: All values defined once in tokens, shared patterns in mixins
- **No runtime cost**: SCSS compiled at build time, CSS custom properties for theming
- **Component isolation**: Each component imports only what it needs
- **Semantic tokens**: Named by purpose (`--color-danger`), not value (`--color-red`)

## 9.9.0 Toasts

Global toast notifications are driven by a tiny store + a single host component:

- `src/lib/stores/toast.ts` — exports `showToast(message, { kind?, duration? })` and a reactive `toasts` store. Kinds: `'info' | 'success' | 'error'`.
- `src/components/Toaster.svelte` — mounted **once** in `src/routes/+layout.svelte`, subscribes to the store, renders stacked toasts, and auto-dismisses each on its duration timer.

Pages never mount their own toaster. Any module — server actions via their client callers, page components, or shared utilities — can call `showToast(...)` and it lands in the global host. Example user: `agents/[id]/edit` fires a success toast on save before redirecting to the inspect page.

## 9.10.0 Confirm Modals

Confirm dialogs (delete agent, delete environment, apply template) use a shared pattern:

- Full-viewport `.confirm-backdrop` (dimmer) with `role="presentation"`
- The backdrop's `onclick` checks `e.target === e.currentTarget` so clicks inside the modal don't close it — no inner `stopPropagation` handler needed
- `onkeydown` handles `Escape` on the backdrop
- Inner `.confirm-modal` has `role="dialog"` and `tabindex="-1"`
- Actions are labeled by outcome ("Delete Agent", "Apply template") — never ambiguous "OK" / "Cancel" alone
