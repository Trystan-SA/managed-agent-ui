# 6.0.0 — Chat & Real-time Streaming

## 6.1.0 Overview

The chat interface provides real-time interaction with Claude agents via Server-Sent Events (SSE). User messages go through REST; agent responses stream back via SSE.

A single component — `ChatView` — owns the entire chat surface (transcript, SSE lifecycle, event→message conversion, auto-scroll, composer, working panel, cancel). Both the dashboard's primary chat surface (see [7.0.0](./7.0.0-dashboard.md)) and the `/chat/[id]` detail page mount `ChatView`; layout is parent-controlled via the `--chat-view-padding` CSS custom property.

## 6.2.0 Communication Pattern

```
User action  →  POST /api/sessions/[id]/events  →  Anthropic API
Agent stream ←  GET  /api/sessions/[id]/stream   ←  Anthropic API (SSE)
```

Bidirectional communication without WebSocket — REST for user actions, SSE for agent responses.

## 6.3.0 Event Types

### 6.3.1 User → Agent (sent via POST)

| Event | Description |
|-------|-------------|
| `user.message` | Text message from user |
| `user.interrupt` | Stop current agent response (see 6.7.0 Cancel) |
| `user.custom_tool_result` | Result for custom tool call |
| `user.tool_confirmation` | Confirm/deny tool execution |

### 6.3.2 Agent → User (received via SSE)

| Event | Description |
|-------|-------------|
| `agent.message` | Text response (streamed token-by-token) |
| `agent.thinking` | Extended thinking content |
| `agent.tool_use` | Tool invocation (name + input) |
| `agent.tool_result` | Tool execution output |
| `agent.mcp_tool_use` | MCP tool invocation |
| `agent.mcp_tool_result` | MCP tool output |
| `agent.custom_tool_use` | Custom tool request |
| `session.status_idle` | Agent finished processing |
| `session.status_running` | Agent started processing |

## 6.4.0 SSE Streaming Implementation

1. Client opens `EventSource` to `/api/sessions/[id]/stream`
2. Server route creates an Anthropic SDK stream
3. Stream is proxied through SvelteKit's `ReadableStream`
4. Client parses SSE events and folds them into the local `messages` array via `$lib/utils/chatEvents.applyEventToMessages`
5. `session.status_idle` closes the `EventSource` client-side

On page load, `ChatView` first hydrates from preloaded history (when the parent passes it) or fetches `/api/sessions/[id]/events`, then opens the live stream.

## 6.5.0 `ChatView` Component

`src/components/ChatView.svelte` is the single chat surface.

### 6.5.1 Responsibilities

- Own the `messages` array and apply incoming events to it
- Manage the `EventSource` lifecycle (open on live session, close on idle/unmount)
- Handle the initial history fetch (unless messages were preloaded)
- Render composer, working panel, and auto-scroll
- Call `/api/sessions/[id]/events` with `user.message` on submit
- Call `/api/sessions/[id]/events` with `user.interrupt` on Cancel
- Notify the parent of status changes and session creation

### 6.5.2 Props

| Prop | Purpose |
|------|---------|
| `sessionId: string \| null` | Active session, or `null` for the new-chat flow |
| `preloadedMessages?` | Skip the initial events fetch when the parent already loaded history server-side |
| `preloadedStatus?` | Seed status (typically from the server-loaded session) |
| `createSession?: () => Promise<ChatSession \| null>` | New-chat flow: factory invoked when the first message is sent with `sessionId === null` |
| `onSessionCreated?`, `onStatusChange?` | Callbacks so the parent can reflect session/state |
| `composerPlaceholder?`, `canSend?` | Composer affordances |
| `emptyState?: Snippet` | Parent-supplied empty state (e.g., "pick an agent to start") |

### 6.5.3 Layout contract

The parent controls outer padding/width cap via `--chat-view-padding`. The dashboard uses a wider cap with more vertical whitespace; `/chat/[id]` uses a tighter, card-framed padding. `ChatView` itself renders no outer chrome.

## 6.6.0 Composer & Working Panel

The composer is **inline at the bottom of the transcript flow** — not a fixed bottom bar. While `status === 'running'`, the composer is replaced by a Working panel containing an animated indicator and a Cancel button. Returning to `idle` swaps the composer back in.

## 6.7.0 Cancel (`user.interrupt`)

The Cancel button posts `user.interrupt` via `/api/sessions/[id]/events`. The stream terminates, `session.status_idle` fires, and the composer returns. No client-side state cleanup is required — the event is authoritative.

## 6.8.0 Message Rendering

`src/components/ChatMessage.svelte` renders each turn.

- **User turns** — right-aligned quiet note with a tracked `YOU` label; no card or avatar. Intentionally recessive so the agent's transcript reads as the foreground.
- **Assistant turns** — transcript style, filled indigo monogram (`A`) badge, readable name header, markdown body, tool/thinking blocks inline.
- **Thinking** — `ThinkingBlock` renders a compact `Thought` marker plus a blockquote-style indented mono body. Empty thought blocks are filtered at render time so they don't leave gaps in the transcript.

## 6.9.0 Supporting Components

| Component | Purpose |
|-----------|---------|
| `ChatMessage` | One turn (user or assistant) — see 6.8.0 |

The standalone `/chat` listing has been folded into `/dashboard` — there is no separate index.

## 6.11.0 Design Decisions

1. **One ChatView, two hosts.** Dashboard and `/chat/[id]` used to duplicate the transcript, composer, and SSE plumbing. `ChatView` is now the only place those exist; layout differences live in the parent via a CSS variable.
2. **SSE over WebSocket.** Agent responses are unidirectional streams; user actions are discrete REST events. SSE is simpler to proxy and debug.
3. **Composer inline, not fixed.** The fixed bottom bar hid content on short viewports and fought the transcript's natural reading flow. Inline composer keeps the last turn and the input in the same scroll context.
4. **Cancel via `user.interrupt`, not a client abort.** The event is the source of truth — trying to short-circuit the stream locally creates inconsistent state between browser and server.
